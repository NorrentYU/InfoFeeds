# datafetch-contracts

## 1. 合同目标
定义 datafetch 模块在“允许终止会话前”必须满足的完成条件（Definition of Done + Exit Gate）。

## 2. 全局退出条件（全部必须满足）
1. 已按固定顺序完成模块：Telegram -> Substack -> YouTube -> X -> Others -> Aggregation。
2. 每个已完成渠道均有测试记录，且包含：
   - Telegram/Substack/YouTube/Others：选取的 2 个信息源（必须来自 `sourceList.md`）。
   - X：通过“CDP 直连专用已登录 Chrome（主路径）”抓取 For You 个性化推送（测试 5 条，生产 20 条）；并保留会话/密码恢复兜底证据。
   - 抓取时间窗口（非 X 默认 24h，若改动需注明）。
   - 原始链接证据与摘要结果证据。
3. 生成的聚合 Markdown 严格符合目标格式，字段齐全：
   - 标题、日期时间、渠道分节、来源名、摘要、原链接。
4. PDF 成功生成，内容与 Markdown 一致，且可打开读取。
5. 聚合层 manifest 成功生成并可追溯（输入数、过滤数、去重数、摘要成功/失败数、失败明细、输出路径）。
6. 有效性校验通过：
   - 摘要内容来自真实正文，不得是 CAPTCHA/防火墙/拦截页文本。
   - Telegram 场景中，若消息为外链，必须读取外链正文后再总结；若无外链降级，需有 `no_external_link` 证据且 `url` 置空。
7. 定时任务配置可工作：
   - 默认时区 UTC+8。
   - 默认推送时间 08:30。
   - 默认时间范围 24h。
8. 管理端能力已覆盖：
   - 增删信息源。
   - 修改推送时间与频次。
   - 修改总结 prompt。
9. 端到端联调通过：从抓取 -> 清洗 -> 总结 -> 聚合 -> PDF -> 本地落盘全链路成功一次。
10. 所有 blocker 均已关闭；如未关闭，必须明确写入“不可终止原因”。

## 3. 渠道级退出条件（每个渠道都要满足）
1. Telegram/Substack/YouTube/Others：从 `sourceList.md` 中选 2 个源进行实测。
2. X：优先使用 CDP 直连专用已登录 Chrome 进行实测；若不可用再走会话/凭证恢复路径。
3. X 测试阶段只要求验证前 5 条个性化推送；正式运行再使用前 20 条。
4. Telegram/Substack/YouTube/Others：在同一时间窗口内抓到至少 1 条有效更新；若无更新需记录证据并重试窗口。
5. X：按抓取时刻返回前 N 条（测试 N=5，生产 N=20），不以时间窗口为通过前提。
6. 每个测试对象至少产出 1 条“摘要 + 链接”；Telegram 无外链降级记录允许 `url` 为空但必须有失败留痕。
7. 摘要质量最小标准：
   - 能表达核心事实/观点。
   - 不少于 80 字（中文）或 60 词（英文）。
   - 不得出现“无法访问页面所以总结失败但仍输出页面拦截文案”。
8. 渠道模块测试通过后方可进入下一个渠道。
9. Others 渠道主路径必须使用 RSS `/feed`；若未使用，必须提供升级批准记录。

## 4. 聚合层退出条件（必须满足）
1. 聚合层开发与测试分别按 `contracts/tasks/aggregation-dev-task.md` 与 `contracts/tasks/aggregation-test-task.md` 执行并通过。
2. 默认 prompt 文件 `contracts/prompts/default-summary-prompt.md` 可加载；用户自定义 prompt 可叠加且不破坏输出结构。
3. 摘要输出格式必须稳定为“`**标题**` + 1-3 段正文”，禁止列表与编号。
4. 跨渠道重复链接必须去重，同 canonical URL 在最终 Markdown 仅出现一次。
5. 摘要失败条目不得伪造成功，必须记录到 manifest。
6. `digest-*.md`、`digest-*.pdf`、`digest-*.manifest.json` 三件套必须同批次生成。

## 5. X 渠道安全与可用性约束
1. 严禁在日志、报告、截图中输出 `.env` 中的账号或密码。
2. 如遇 2FA、风控验证、人机校验，记录阻塞证据并上报，不得伪造通过。
3. X 原链接允许“外链原文链接”或“推文状态链接”，但不得是聚合页或跳转壳链接。
4. X 渠道采用非官方网页自动化路径，页面结构变化导致失败时必须按阻塞流程处理。
5. X 渠道必须支持“同会话人工接管挑战 -> 会话持久化复用”流程。
6. X 渠道必须在独立专用浏览器 profile 中运行，不得复用用户日常浏览器会话。
7. X 渠道必须具备 stale feed 自动刷新重试机制（最多 3 次）。
8. X 渠道必须具备 CDP 可用性闸门：`x:cdp-smoke` 在测试环境可稳定返回前 5 条或给出明确阻塞证据（如 `ECONNREFUSED`、未登录态）。

## 6. 阻塞与升级规则
1. 除 X 外，`sourceList.md` 中某渠道少于 2 个可用源：立刻阻塞并上报，不允许标记完成。
2. 站点反爬导致正文不可得：必须切换备选源重试；仍失败则记录失败证据并上报。
3. 格式校验失败或 PDF 渲染失败：必须修复后回归，不允许带缺陷结束会话。
4. X 登录态失效或账号触发安全挑战：标记为 X 专项阻塞并上报。
5. X 人工接管流程不可用或会话无法持久化：标记为 X 专项阻塞并上报。
6. CDP 端口不可用、上下文为空或专用 Chrome 未登录：标记为 X 专项阻塞并上报。
7. LLM 调用连续失败导致摘要成功率低于 90%：标记为聚合层阻塞并上报。
8. Markdown 与 PDF 内容不一致：标记为聚合层阻塞并上报。

## 7. 当前已知前置风险
1. X 渠道依赖真实登录态与平台风控状态，存在被 2FA/验证码阻断的风险。
2. 若出现平台风控阻塞，项目可继续推进其他渠道，但整体最终验收需等待 X 恢复。
