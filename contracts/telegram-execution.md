# Telegram Module Execution Pack

## 1. 模块目标
在最近 24h 时间窗口内，从指定 Telegram 频道抓取更新，优先解析消息中的原文外链并抽取正文；无外链但消息正文有效时可降级入库（`url` 置空并留痕）；通过格式与有效性测试后，进入下一模块。

## 2. 模块范围
1. 本轮只做 Telegram 渠道，不触发 Substack/YouTube/X/Others 开发。
2. 固定测试源：
   - `https://t.me/cookiesreads`
   - `https://t.me/web3list`
3. 处理公开可访问消息与其外链正文；无外链消息允许按降级策略处理。

## 3. 开发单（给开发 agent）
开发 agent 执行文件：`contracts/tasks/telegram-dev-task.md`

### 3.1 输入
1. `sourceList.md` 的 Telegram 分组。
2. 调度参数（默认 24h，UTC+8 08:30）。
3. 用户摘要 prompt（由设置层注入）。

### 3.2 实现任务
1. 构建 Telegram 抓取器：获取窗口内消息列表。
2. 实现消息外链解析器：提取并规范化外链 URL。
3. 实现正文提取器：抓取外链正文并做可读性清洗；支持 Substack RSS 与 X 外链兜底提取。
4. 实现有效性过滤器：拦截 CAPTCHA/防火墙/空正文。
5. 无外链降级策略：若消息正文通过校验，可入库并记录 `no_external_link` 失败项（`url` 置空）。
6. 输出统一中间结构：
   - `source_type=telegram`
   - `source_name`
   - `title`
   - `content`
   - `url`（原文链接；无外链降级时允许为空）
   - `published_at`
   - `fetched_at`
7. 失败留痕：记录失败类型与重试结果（脱敏），包括 `no_external_link` 降级证据。

### 3.3 禁止事项
1. 不得把 Telegram 消息链接当作最终 `url` 输出（降级记录除外，降级时 `url` 必须为空）。
2. 不得把拦截页文本送入摘要层。
3. 不得绕过统一字段契约。

### 3.4 交付物
1. Telegram 渠道标准化数据样例（至少覆盖两个测试源）。
2. 开发自测记录（含失败重试案例）。
3. 已知限制列表（若有）。

## 4. 测试单（给测试 agent）
测试 agent 执行文件：`contracts/tasks/telegram-test-task.md`

### 4.1 通用用例
1. 时间窗口过滤正确（默认最近 24h）。
2. 频道消息抓取成功且可重复执行。
3. 外链抽取正确，非外链内容不误判。
4. 正文提取有效，空正文/异常正文可识别。
5. 输出字段完整且符合统一结构。

### 4.2 Telegram 专项用例
1. 每个测试源至少 1 条有效摘要输入（共至少 2 条）。
2. 外链记录的 `url` 必须是原文链接，不是 Telegram 消息链接。
3. 至少验证 1 条“无外链降级”样例：`content` 来自消息正文、`url` 为空、并有 `no_external_link` 失败留痕。
4. 至少验证 1 个异常页面样例（CAPTCHA/拦截页）被正确拦截。
5. 结果能按目标模板落入 `Telegram` 分节。

### 4.3 通过标准
1. 所有用例通过。
2. 无 P0/P1 未关闭缺陷。
3. 有完整测试证据可追溯。

## 5. 验收闸门（架构师）
1. 开发交付物齐全且符合契约。
2. 测试报告通过且证据完整。
3. Telegram 结果可被下游摘要与聚合层直接消费。
4. 仅在上述 1-3 全部满足时，允许切换到 Substack 模块。

## 6. 阻塞升级
1. 连续 2 次抓取失败：升级架构师。
2. 外链正文普遍受限（反爬/区域限制）：升级架构师决定备选策略。
3. 契约字段不一致：先冻结新契约，再继续开发。
