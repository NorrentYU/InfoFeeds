# Coding Rules

## 1. 通用约束
1. 只实现当前 task contract 明确要求的范围，不扩展需求。
2. 不得把账号、密码、token 写入代码、日志、测试快照、示例文件。
3. 配置必须外置（环境变量或配置文件），禁止硬编码。
4. 新增逻辑优先模块化，避免在单文件堆叠复杂流程。
5. 变更必须可回归：每个功能点对应至少一条可执行测试。
6. 模块迭代时必须先做第一性原理拆解，再重构关键路径；禁止“在旧实现上加一层判断”的补丁堆叠。
7. 若现有结构已不适配新需求，优先重写模块内聚边界（接口、状态机、错误模型），再迁移实现。

## 2. 数据契约约束
1. 统一中间结构字段固定为：`source_type/source_name/title/content/url/published_at/fetched_at`。
2. 若上游数据缺字段，允许置空，但必须记录失败原因或降级原因。
3. `url` 必须是原文链接；Telegram 消息链接仅用于追踪，不用于最终展示。
4. `content` 必须是可读正文，禁止写入验证码页/风控页文案。

## 3. Telegram 模块实现约束
1. 先抓 Telegram 频道消息，再解析消息内外链并抓取外链正文。
2. 去重策略至少覆盖：同一原文 URL、同一消息重复转发。
3. 时间窗口默认 24h，支持由调度参数覆盖。
4. 无外链但正文有效时允许降级入库（`url` 置空），并同步记录 `no_external_link` 失败留痕。
5. 抓取失败要保留可诊断信息（源、时间、错误类型），但不保留敏感信息。

## 4. Substack 模块实现约束
1. 主路径固定使用 `<source>/feed`（RSS）。
2. 内容字段优先级固定：`content:encoded` > `description`。
3. 去重策略固定：`guid` 优先，缺失时 `link + published_at`。
4. 最终 `url` 必须为原文链接，不能是 feed 链接。
5. 无效内容（空内容/订阅墙文案/异常页文案）必须过滤，不进入摘要层。

## 5. YouTube 模块实现约束
1. 默认窗口固定为 `T-48h ~ T-24h`（除非架构师另行冻结新策略）。
2. 文本获取策略固定：仅使用可用字幕（字幕-only）。
3. `url` 必须是具体视频链接，禁止频道页链接入库。
4. 每条成功记录必须标记文本来源（`caption`）。
5. 无字幕项必须标记 `transcript_missing` 并留痕，不得静默吞错。

## 6. Others（newsletter）模块实现约束
1. 主路径固定使用 `<source>/feed`（RSS），与 Substack 同构。
2. 内容字段优先级固定：`content:encoded` > `description`。
3. 去重策略固定：`guid` 优先，缺失时 `link + published_at`。
4. 最终 `url` 必须为原文链接，不能是 feed 链接。
5. 无效内容（空内容/订阅墙文案/异常页文案）必须过滤，不进入摘要层。
6. `/feed` 不可用时必须先记录阻塞证据，再升级，不得静默切 HTML 主路径。

## 7. X 模块实现约束
1. 数据来源固定为登录态下的 `For You` 流，不读取 `sourceList.md` 的 X 配置。
2. 抓取条数固定支持：测试 5 条，生产 20 条。
3. 抓取前必须确认当前标签为 `For You`，不是 `Following`。
4. `url` 优先使用外链原文链接，未命中时回退推文状态链接；禁止聚合页/跳转壳链接入库。
5. 凭证与会话敏感信息必须严格脱敏，禁止写入日志和报告。
6. 遇到登录挑战/风控校验时必须留痕并中止，禁止绕过或伪造结果。
7. 认证流程优先级固定为“CDP 直连专用已登录 Chrome -> 本地会话复用 -> 密码/人工接管兜底”，定时任务中禁止每次使用账号密码直登。
8. CDP 路径必须显式处理：端口不可达、无上下文、未登录 For You 三类失败，不得混淆成统一登录失败。
9. 人工接管挑战必须在受控专用 profile 中完成，完成后持久化会话供后续复用。
10. 必须使用独立专用浏览器 profile，禁止复用用户日常浏览器档案。
11. 若抓取结果整体超过 24h，先自动刷新重试（最多 3 次）再判定 `stale_feed`。
12. 单条推文状态链接抓取只允许一次尝试，禁止在模块内对同一推文做循环重试打开关闭。

## 8. 可维护性约束
1. 复杂分支前允许添加简短注释说明意图。
2. 错误处理必须区分可重试与不可重试。
3. 对外接口（函数/模块）要保持输入输出稳定，避免隐式副作用。
4. 每次重构后必须删除已废弃分支与死代码，避免双轨逻辑长期并存。

## 9. 聚合层实现约束
1. 聚合层输入只接受统一字段记录，不得引入渠道私有字段作为必填依赖。
2. 聚合层必须执行二次正文有效性校验，命中拦截页关键词的内容禁止送入摘要模型。
3. 跨渠道去重键固定为 canonical URL；冲突时优先保留正文更完整的记录。
4. 摘要 prompt 必须以默认 prompt 为基底，用户 prompt 仅做追加约束，不得替换默认结构。
5. 摘要输出必须严格解析为“`**标题**` + 1-3 段正文”，不合规即判失败并留痕。
6. 最终分节顺序固定：Telegram -> X -> Substack -> Youtube -> 其他。
7. 必须产出 `digest-*.md`、`digest-*.pdf`、`digest-*.manifest.json` 三件套且版本一致。
